# 对软件设计的总结

2020.1.19

Spring Cloud从2017年开始在项目中使用，所以微服务这套体系还算比较熟悉。

但是经常看到别人提的“设计模式”、“领域驱动设计”又是怎么回事呢？

自己理解：狭义的设计模式就是GoF的23种设计模式（设计模式笔记），广义的设计模式还应该包括像MVC、前后端分离CSR（Controller+Service+Repository）、netty使用的Reactor模式、微服务中的断路器模式等等，是各种优秀的、可以重复使用的经验。

最早接触设计模式是在读研时选修《面向对象编程》课程的时候，那时心思还在信息安全专业上，项目中用的是C语言，那个课没怎么听明白。

接着就是2017年，那时候从信息安全转Java开发有一两年了，花时间阅读了《head first设计模式》，读得过瘾，但是在之后的项目实践中基本没有用上。反倒是在一次面试中考到了装饰者模式，题目和书中案例是一模一样的。也大概是那时候，从一个做了很多年研发的朋友那里首次听到了领域驱动设计（DDD），在烽火星空公司的内网资料里，读到了《人月传说》。记得《人月传说》中说过的一个观点：并不是人越多，软件就能做越快越好，10个人做10个月的软件，并不能通过把人加到100而1个月就完成，10来个人的团队效率往往最高，大的团队效率可能下降。

18年加入瀚云，参与搭建工业互联网平台，接触到了高并发（7k TPS）、大数据（1个月100亿数据）的业务场景，采用的是Spring Cloud的微服务架构，微服务划分比较清晰（大部分按业务领域划分），但对DDD的思想应用还是不多。在具体的微服务开发时，仍然是数据表驱动式（SQL-Driven）的开发，拿到原型设计后就开始数据库表设计，再接着mysql的增删查改的接口开发，以此往上，直到UI界面的开发和前后端的联调。在项目中，用mybatis generator生成了Repository和Entity，自己写了Controller和VO，但是在Service一层没有BO，所有业务逻辑都写在Service中，是典型的贫血模型开发模式。由于瀚云的业务不复杂，贫血模型开发模式已经足够了。

19年，当我在极客时间上学习了《设计模式之美》这门课以后（特别是第12讲），才算真正理解了充血模型。也是在这一年，在C语言中文网Java设计模式复习了一遍设计模式，有温故知新、豁然开朗的感觉。

对于DDD，现目前没有想过去啃Eric Evans大神的书，最多会看看《领域驱动设计精简版》，这个小书的前面两章节“何为领域驱动设计”、“通用语言”看着过瘾、有共鸣。第一章讲的是做软件开发首先要理解需求，掌握领域知识，用模型来交流。第二章讲在和领域专家沟通的时候要创建一种模型为主的通用语言，个人理解不止是研发和领域专家，诸如研发和产品之间，研发内部之间都应该用这种模型语言，或者通俗点讲，应该把概念和术语明确下来。剩下的章节我觉得概念太多，和实际项目差别比较多，只是泛泛而读，这些知识会不会用上看未来的缘分吧。

现在，内心很有一种想实践的冲动，想要合理的运用设计模式、充血模型去优雅的实现一些复杂的业务，希望未来能有机会。

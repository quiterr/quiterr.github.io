# 对项目管理的总结

2020.1.29

# 敏捷开发

最早学习敏捷开发是2017年，看了这篇博文敏捷开发之Scrum扫盲篇，至今日能记得的东西不多了，但在项目实践中一直采用的是快速迭代的方式，在瀚云大概每周会做两次发布上线，使用禅道来做项目管理。

# 团队管理

在微服务架构场景下，一个大的项目被分成多个小的微服务，在瀚云我有5个做Java后端开发的兄弟，通常我像这样分配任务：

任意一个微服务都有至少两个人参与，这符合配对开发的理念，两个人可以互相讨论、做CR，互为备份，当一个人休假或离职时，项目整体不受影响。一个人会参与3-5个微服务，过少工作量不充分，过多则切换成本高。

听过一个阿里38号员工，HR总监讲的阿里的成长过程，其中一节讲到了团队管理。只有当“工作满意”、“钱”、“个人发展”三方面的需求得到合理的满足时，员工的忠诚度才会高。而不同时期的员工又会在这三方面有不同的侧重，比如有的人就想要轻松安逸，团队和谐，对前途和钱途看得不那么重，刚工作的人看重能不能学到知识，工作了三四年的人要买房成家可能最看重薪酬，这些都需要leader平时多了解自己的下属，才能恰当的激励下属。

# 项目管理

在中移物联网做信息安全的时候，公司就组织过项目管理的专项培训，培训的小书我一直留在家里书架上。这几年转做研发，有的项目是产品经理兼项目经理，我确实感受到了不专业的项目经理存在的问题，比如没有理清项目干系人，项目不兼容上线的时候没有提前通知到对方，导致对方的应用出现故障。

项目中可能出现的各种问题，例如：

  • 缺乏技术或技术不全面
  • 有人退出项目组
  • 团队成员之间过于友好（闲聊、讨论私人问题）
  • 顽固的团队（不愿尝试新方法，坚持认为自己的方法是唯一正确的）
  • 团队内部矛盾
  • 把时间浪费在不重要的任务上
  • 工作质量很差（误解工作要求、衡量工作标准不同、不具备所需的技能）
  • 筋疲力竭（比如同时参与多个项目）

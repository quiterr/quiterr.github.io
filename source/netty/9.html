# keepalive 和 idle

keepalive是为了保持设备和平台之间的连接，但如果接入的设备很多，比如100万个，那keepalive的数据包也会非常多。实践中一般是idle时才发keepalive数据包，减少不必要的网络通信。

# keepalive

Server 端开启 TCP keepalive：

bootstrap.childOption(ChannelOption.SO_KEEPALIVE,true)
bootstrap.childOption(NioChannelOption.of(StandardSocketOptions.SO_KEEPALIVE), true)

进到childOption方法中去，可以知道参数是设置到了childOptions这个成员变量中，如果在该类中搜索childOptions就会发现赋值给了currentChildOptions。再跟下去，又传给了内部类ServerBootstrapAcceptor，这是用来处理网络连接的类，它的channelRead方法调了setChannelOptions(child, childOptions, logger);。接着跟进去，会发现!channel.config().setOption((ChannelOption<Object>) option, value)，这里出现了ChannelConfig概念。

在NioServerSocketChannelConfig类，找到setOption：

@Override
public <T> boolean setOption(ChannelOption<T> option, T value) {
    if (PlatformDependent.javaVersion() >= 7 && option instanceof NioChannelOption) {
        return NioChannelOption.setOption(jdkChannel(), (NioChannelOption<T>) option, value);
    }
    return super.setOption(option, value);
}

两种设置keepalive的区别在上面这段代码就可以看出来了，非NioChannelOption，会调super.setOption(option, value);。总之，还是建议用NioChannelOption。

再跟进去就会发现调了JDK NIO的API来设置option，那为什么不直接用JDK呢？注释写得挺清楚了：

// It's important to not use java.nio.channels.NetworkChannel as otherwise the classes that sometimes call this
// method may not be used on Java 6, as method linking can happen eagerly even if this method was not actually
// called at runtime.
//
// See https://github.com/netty/netty/issues/8166

// Internal helper methods to remove code duplication between Nio*Channel implementations.
@SuppressJava6Requirement(reason = "Usage guarded by java version check")
static <T> boolean setOption(Channel jdkChannel, NioChannelOption<T> option, T value) {
    java.nio.channels.NetworkChannel channel = (java.nio.channels.NetworkChannel) jdkChannel;
    if (!channel.supportedOptions().contains(option.option)) {
        return false;
    }
    if (channel instanceof ServerSocketChannel && option.option == java.net.StandardSocketOptions.IP_TOS) {
        // Skip IP_TOS as a workaround for a JDK bug:
        // See http://mail.openjdk.java.net/pipermail/nio-dev/2018-August/005365.html
        return false;
    }
    try {
        channel.setOption(option.option, value);
        return true;
    } catch (IOException e) {
        throw new ChannelException(e);
    }
}

# Idle

开启Idle Check:

ch.pipeline().addLast("idleCheckHandler", new IdleStateHandler(0, 20, 0, TimeUnit.SECONDS));

创建IdleStateHandler的几个参数，看命名就很清晰了：

/**
    * @see #IdleStateHandler(boolean, long, long, long, TimeUnit)
    */
public IdleStateHandler(
        long readerIdleTime, long writerIdleTime, long allIdleTime,
        TimeUnit unit) {
    this(false, readerIdleTime, writerIdleTime, allIdleTime, unit);
}

IdleStateHandler的源码在io.netty.handler.timeout包下，netty的注释普遍写得比较好，不仅描述该类干什么的，还有使用例子。在该类的initialize方法中可以看到：

lastReadTime = lastWriteTime = ticksInNanos();
if (readerIdleTimeNanos > 0) {
    readerIdleTimeout = schedule(ctx, new ReaderIdleTimeoutTask(ctx),
            readerIdleTimeNanos, TimeUnit.NANOSECONDS);
}
if (writerIdleTimeNanos > 0) {
    writerIdleTimeout = schedule(ctx, new WriterIdleTimeoutTask(ctx),
            writerIdleTimeNanos, TimeUnit.NANOSECONDS);
}
if (allIdleTimeNanos > 0) {
    allIdleTimeout = schedule(ctx, new AllIdleTimeoutTask(ctx),
            allIdleTimeNanos, TimeUnit.NANOSECONDS);
}

也就是定时执行ReaderIdleTimeoutTask、WriterIdleTimeoutTask、AllIdleTimeoutTask，这几个都是内部类。对于ReaderIdleTimeoutTask，我在代码里边加了自己的注释：

        @Override
        protected void run(ChannelHandlerContext ctx) {
            // 假如idle检查时间是60秒，当前时间减去上一次读的时间是5秒，那下一次做idle检查的时间是55秒后
            // nextDelay = lastReadTime + readerIdleTimeNanos - ticksInNanos() 这样写就更清楚了
            long nextDelay = readerIdleTimeNanos;
            if (!reading) {
                nextDelay -= ticksInNanos() - lastReadTime;
            }

            // 假如idle检查时间是60秒，当前时间减去上一次读的时间是65秒，nextDelay就会为负，说明是read idle
            if (nextDelay <= 0) {
                // Reader is idle - set a new timeout and notify the callback.
                readerIdleTimeout = schedule(ctx, this, readerIdleTimeNanos, TimeUnit.NANOSECONDS);
                // 第一次idle会特殊对待
                boolean first = firstReaderIdleEvent;
                firstReaderIdleEvent = false;

                try {
                    // 发出一个idle事件
                    IdleStateEvent event = newIdleStateEvent(IdleState.READER_IDLE, first);
                    channelIdle(ctx, event);
                } catch (Throwable t) {
                    ctx.fireExceptionCaught(t);
                }
            } else {
                // Read occurred before the timeout - set a new timeout with shorter delay.
                readerIdleTimeout = schedule(ctx, this, nextDelay, TimeUnit.NANOSECONDS);
            }
        }
    }

要注意channelIdle(ctx, event);这行代码，这个方法在IdleStateHandler中有实现，在ReadTimeoutHandler中复写了，前者是发出事件，后者是发出ReadTimeoutException异常。

WriterIdleTimeoutTask唯一不同的地方就是当判断为idle的时候，多了一个判断：

if (hasOutputChanged(ctx, first)) {
    return;
}

当observeOutput为true时，检查两次调hasOutputChanged方法之间是否有变化。个人理解这是因为缓冲区在写入满之后网卡才会把数据发出去，如果缓冲区在变，也可以认为write没有idle。

AllIdleTimeoutTask的逻辑和前述两个类似。须注意io.netty.handler.timeout包下的WriteTimeoutHandler，它和ReadTimeoutHandler不一样，不是用来做idle检查的，是用来做写入超时检查的。

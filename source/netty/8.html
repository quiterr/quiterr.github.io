# 粘包&半包

粘包产生的原因：

  • 发送方发送的数据比较小，网卡缓冲区会积累到一定的大小才会真的发送。
  • 接收方读取缓冲区的数据不够及时

半包产生的原因：

  • 发送方发送的数据比较大，超过网卡缓冲区的大小
  • 发送的数据大于MTU的大小

粘包的另外两种说法：

  • 多个发送被一次接收-收发角度
  • 多个发送共用了一个数据包-传输角度

半包的另外两种说法：

  • 一次发送被多次接收-收发角度
  • 一次发送分成了多个数据包-传输角度

# 从源码剖析解码逻辑

上图中的三个Decoder都继承自ByteToMessageDecoder，后者又继承自ChannelInboundHandlerAdapter，核心逻辑在channelRead方法中。

进到channelRead，可以知道：收到数据后，先用数据积累器积累数据，之后调callDecode方法开始解码。

进到callDecode，可以发现主逻辑在decodeRemovalReentryProtection();中。

进到decodeRemovalReentryProtection，发现主逻辑在：

protected abstract void decode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out) throws Exception;

这是一个抽象方法，显然这是用模板方法模式来把处理逻辑交给子类，利用idea的功能，很容易找到实现，例如FixedLengthFrameDecoder的实现为：

protected Object decode(
        @SuppressWarnings("UnusedParameters") ChannelHandlerContext ctx, ByteBuf in) throws Exception {
    if (in.readableBytes() < frameLength) {
        return null;
    } else {
        return in.readRetainedSlice(frameLength);
    }
}

再来看看之前说的数据积累器，首先它的定义为：

private Cumulator cumulator = MERGE_CUMULATOR;

再者，从channelRead调用的地方跟进去:

public interface Cumulator {
    ByteBuf cumulate(ByteBufAllocator alloc, ByteBuf cumulation, ByteBuf in);
}

这个接口有两种实现，分别是MERGE_CUMULATOR、COMPOSITE_CUMULATOR，两个都在ByteToMessageDecoder中，其中前者是memory copy的实现，也是ByteToMessageDecoder默认的选择。

# channel注册

上一篇EchoServer创建好了bossGroup(main reactor)和workerGroup(sub reactor)，接着就是：

EventLoopGroup bossGroup = new NioEventLoopGroup(1);
EventLoopGroup workerGroup = new NioEventLoopGroup();
final EchoServerHandler serverHandler = new EchoServerHandler();
try {
    ServerBootstrap b = new ServerBootstrap();
    b.group(bossGroup, workerGroup)
// 后面的代码省略

从这里跟进去，可以知道bossGroup被设置到了AbstractBootstrap的group成员变量：

volatile EventLoopGroup group;

通过idea的find usage功能可以找到哪些地方在访问这个group变量

从return group;这句可以去到

@Deprecated
public final EventLoopGroup group() {
    return group;
}

在进一步，用idea的call hierarchy功能可以看哪些地方在调用group()方法

每个都点开看，很容易知道关键代码是AbstractBootstrap.initAndRegister()：

ChannelFuture regFuture = config().group().register(channel);

从这里就可以看出来了，这里把channel（服务端是ServerSocketChannel）注册到了bossGroup。

再看workerGroup，从b.group(bossGroup, workerGroup)跟进去，知道名字变成了childGroup，找哪里在用childGroup就可以到关键的ServerBootstrap.init()方法：

@Override
void init(Channel channel) {
    setChannelOptions(channel, options0().entrySet().toArray(EMPTY_OPTION_ARRAY), logger);
    setAttributes(channel, attrs0().entrySet().toArray(EMPTY_ATTRIBUTE_ARRAY));

    ChannelPipeline p = channel.pipeline();

    final EventLoopGroup currentChildGroup = childGroup;
    final ChannelHandler currentChildHandler = childHandler;
    final Entry<ChannelOption<?>, Object>[] currentChildOptions =
            childOptions.entrySet().toArray(EMPTY_OPTION_ARRAY);
    final Entry<AttributeKey<?>, Object>[] currentChildAttrs = childAttrs.entrySet().toArray(EMPTY_ATTRIBUTE_ARRAY);

    p.addLast(new ChannelInitializer<Channel>() {
        @Override
        public void initChannel(final Channel ch) {
            final ChannelPipeline pipeline = ch.pipeline();
            ChannelHandler handler = config.handler();
            if (handler != null) {
                pipeline.addLast(handler);
            }

            ch.eventLoop().execute(new Runnable() {
                @Override
                public void run() {
                    pipeline.addLast(new ServerBootstrapAcceptor(
                            ch, currentChildGroup, currentChildHandler, currentChildOptions, currentChildAttrs));
                }
            });
        }
    });
}

最后的一行代码把currentChildGroup传给了ServerBootstrapAcceptor

pipeline.addLast(new ServerBootstrapAcceptor(
    ch, currentChildGroup, currentChildHandler, currentChildOptions, currentChildAttrs));

继续往下是ServerBootstrap.channelRead()，将child这个SocketChannel注册到了childGroup

childGroup.register(child).addListener(new ChannelFutureListener() {
    @Override
    public void operationComplete(ChannelFuture future) throws Exception {
        if (!future.isSuccess()) {
            forceClose(child, future.cause());
        }
    }
});

总结起来就是两种channel注册到两种group中。

# 主reactor是单线程

在EchoServer中启动server的代码是ChannelFuture f = b.bind(PORT).sync();。结合调用图，可以知道bind()方法最终将ServerSocketChannel注册到了bossGroup。

bind()只会调一次，所以ChannelFuture regFuture = config().group().register(channel);也只会调一次，查看register()的代码：

@Override
public ChannelFuture register(Channel channel) {
    return next().register(channel);
}

只是注册到一个线程而已，所以是单线程的。

# netty中的适配器模式

以下这段代码是上文提过的：

pipeline.addLast(new ServerBootstrapAcceptor(
    ch, currentChildGroup, currentChildHandler, currentChildOptions, currentChildAttrs));

ServerBootstrapAcceptor是ServerBootstrap的内部类：

private static class ServerBootstrapAcceptor extends ChannelInboundHandlerAdapter {
    //省略内部代码
}

ChannelInboundHandlerAdapter是一个ChannelInboundHandler的实现类，这是一个用来处理“输入事件”的类。

public class ChannelInboundHandlerAdapter extends ChannelHandlerAdapter implements ChannelInboundHandler {
    // 省略实现类
}

ServerBootstrapAcceptor的完整类图：

对照适配器模式中的类图，可以知道netty用的是类适配器模式，适配者是ChannelHandlerAdapter，目标接口是ChannelInboundHandler，适配器是ChannelInboundHandlerAdapter，客户端是ServerBootstrap。

从结构上来说，这里确实是适配器模式的结构，但是却没有看出来“接口转换”，只是从调用关系可以知道适配器的继承类ByteToMessageDecoder一定会调适配者的ensureNotSharable();方法。

总结：教科书中适配器模式举例，一般都是两孔插座转三孔插座，存在明显的接口转换。真实场景中的适配器模式有时候并没有明显的接口转换，这时要通过类结构来判断。而在netty的这个应用中，主要是用适配器模式来控制那些没有@Sharable标记的ChannelHandler不被多次加到ChannelPipeline中。

# ChannelHandler详解

# 子类

处理IO event或intercepts IO操作，并转发event到ChannelPipeline的下一个ChannelHandler。

ChannelHandler本身没有提供很多方法，但一般要实现一个它的子类：

  • ChannelInboundHandler处理流入的IO事件
  • ChannelOutboundHandler处理流出的IO操作

另外还提供了如下适配器：

  • ChannelInboundHandlerAdapter处理流入的IO事件
  • ChannelOutboundHandlerAdapter处理流出的IO操作
  • ChannelDuplexHandler可以处理流入和流出的IO事件

# 状态管理

一个ChannelHandler关联一个ChannelHandlerContext。ChannelHandler通过上下文对象，与它所属的ChannelPipeline交互。通过上下文对象，ChannelHandler可以选择上行或下行任意一个方向来传递event，动态修改管道，或通过AttributeKey存储特殊的信息。

ChannelHandler一般都需要存储一些状态信息，最简单也是推荐的做法是使用成员变量。

public interface Message {
    // your methods here
}
 
public class DataServerHandler extends SimpleChannelInboundHandler<Message> {
    private boolean loggedIn;

    @Override
    public void channelRead0(ChannelHandlerContext ctx, Message message) {
        if (message instanceof LoginMessage) {
            authenticate((LoginMessage) message);
            loggedIn = true;
        } else (message instanceof GetDataMessage) {
            if (loggedIn) {
                ctx.writeAndFlush(fetchSecret((GetDataMessage) message));
            } else {
                fail();
            }
        }
    }
 }

ChannelHandler的实例有一个标识connection的状态变量，对每一个新的Channel都必须创建新的ChannelHandler的实例，以避免线程竞争。

 public class DataServerInitializer extends ChannelInitializer<Channel> {
    @Override
    public void initChannel(Channel channel) {
        channel.pipeline().addLast("handler", new DataServerHandler());
    }
}

尽管推荐使用成员变量来存储状态信息，但有时候并不想创建多个ChannelHandler实例，这时可以用ChannelHandlerContext提供的AttributeKey：

public interface Message {
     // your methods here
 }

  @Sharable
 public class DataServerHandler extends SimpleChannelInboundHandler<Message> {
     private final AttributeKey<Boolean> auth =
           AttributeKey.valueOf("auth");

      @Override
     public void channelRead(ChannelHandlerContext ctx, Message message) {
         Attribute<Boolean> attr = ctx.attr(auth);
         if (message instanceof LoginMessage) {
             authenticate((LoginMessage) o);
             attr.set(true);
         } else (message instanceof GetDataMessage) {
             if (Boolean.TRUE.equals(attr.get())) {
                 ctx.writeAndFlush(fetchSecret((GetDataMessage) o));
             } else {
                 fail();
             }
         }
     }
     ...
 }

现在ChannelHandler的状态已经被添加到了ChannelHandlerContext，可以添加多个相同的ChannelHandler实例到不同的pipelines。

# @Sharable注解

io.netty.example.echo.EchoServerHandler就是一个@Sharable标记的类，@Sharable注解是在ChannelHandler中定义，但相关代码在ChannelHandlerAdapter中：

protected void ensureNotSharable() {
    if (isSharable()) {
        throw new IllegalStateException("ChannelHandler " + getClass().getName() + " is not allowed to be shared");
    }
}

public boolean isSharable() {
    Class<?> clazz = getClass();
    Map<Class<?>, Boolean> cache = InternalThreadLocalMap.get().handlerSharableCache();
    Boolean sharable = cache.get(clazz);
    if (sharable == null) {
        sharable = clazz.isAnnotationPresent(Sharable.class);
        cache.put(clazz, sharable);
    }
    return sharable;
}

这些代码我省略了注释，读起来并不难，isSharable()方法就是通过反射获取类是否有@Sharable注解。对于ByteToMessageDecoder这种不能多次加到ChannelPipline的ChannelHandler，在构造函数中就会调ensureNotSharable()，如果程序员在ByteToMessageDecoder类上加了@Sharable注解，就会提示错误了。

总结：@Sharable就是用来标记某个类是否可以多次加入到ChannelPipline。
